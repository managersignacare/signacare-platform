# Runbook — AI Lane Failover

**Purpose:** Phase 4 + 5 — explicit lane selection between the Azure
private fast lane (`azure_fast`), the sovereign GPU lane (`sovereign_gpu`),
and the local Ollama dev lane (`local_ollama`). This runbook covers the
operator decision flow when one lane goes unhealthy.

## 1 — Lane policy summary

| Lane | When healthy | When unhealthy |
|---|---|---|
| `azure_fast` | First choice when configured. <2 s TTFT, cached_tokens telemetry, AAD + Key Vault secrets. | Fail over to `sovereign_gpu` if provisioned; else `local_ollama` (dev/last-resort). |
| `sovereign_gpu` | Compliance-heavy tenants. ~4 s TTFT, model-manifest-pinned, in-tenant VNet. | Fail over to `azure_fast` only after operator sign-off (compliance impact). |
| `local_ollama` | Dev + single-host posture. | No automatic failover — this lane is the bottom of the stack. |

## 2 — Detecting unhealth

The `/api/v1/ai/capabilities` endpoint reports per-lane `health` as one
of `healthy | degraded | unhealthy | disabled`. The signals:

| Signal | Source | Healthy condition |
|---|---|---|
| Endpoint reachable | `azure_fast`: smoke generation through the configured alias over the private endpoint. `sovereign_gpu`: AKS readiness probe at `/api/tags`. | Successful response within lane SLO |
| Deployment exists | Account `deployment-name` resolves | response payload includes `model.version` matching pinned |
| Cached tokens telemetry returning | last 1 hour of `llm_interactions` includes `cached_prompt_tokens >= 0` | non-zero count |
| TTFT | App Insights P50 latency on first token | p50 ≤ `ttftSloMs` |

When ANY lane is `degraded` or `unhealthy`, the AppInsights alert "AI
lane unhealthy" fires. The on-call clinician informatics lead receives
the page.

## 3 — Manual failover procedure

### From `azure_fast` → `sovereign_gpu`

```bash
# 1. Confirm sovereign_gpu lane is healthy via /api/v1/ai/capabilities.
curl -sf -H "Authorization: Bearer $TOKEN" \
  "$API_URL/api/v1/ai/capabilities" \
  | jq '.lanes[] | select(.lane=="sovereign_gpu")'

# 2. Switch the clinic AI runtime backend.
psql "$DATABASE_URL" <<'SQL'
BEGIN;
SELECT id, setting_value FROM staff_settings
  WHERE setting_key = 'ai_runtime'
    AND staff_id IN (SELECT id FROM staff WHERE clinic_id = '<clinic_id>');
UPDATE staff_settings
  SET setting_value = jsonb_set(setting_value, '{llmBackend}', '"local_ollama"', true)
  WHERE setting_key = 'ai_runtime'
    AND staff_id IN (SELECT id FROM staff WHERE clinic_id = '<clinic_id>');
COMMIT;
SQL

# 3. Verify the active lane on the next /capabilities call.
curl -sf -H "Authorization: Bearer $TOKEN" \
  "$API_URL/api/v1/ai/capabilities" | jq '.activeLane.lane'
# Expect: "sovereign_gpu"

# 4. Audit row written for compliance.
psql "$DATABASE_URL" <<'SQL'
INSERT INTO audit_log (id, clinic_id, user_id, action, resource, details, created_at)
VALUES (gen_random_uuid(), '<clinic_id>', '<oncall_staff_id>',
        'ai_lane_failover', 'ai_runtime',
        jsonb_build_object('from','azure_fast','to','sovereign_gpu','reason','<incident_id>'),
        now());
SQL
```

### From `azure_fast` → `local_ollama` (dev / last-resort)

Same procedure but the audit row should note the reason as
"sovereign_gpu unavailable" or "dev environment". `local_ollama` is NOT
a clinical-safety-validated lane; this path is for dev environments and
short-notice operator workarounds only.

### From `sovereign_gpu` → `azure_fast`

Requires operator sign-off due to compliance impact (the sovereign lane
keeps PHI on-tenant; `azure_fast` sends scrubbed prompts to Azure
OpenAI under BAA). Sign-off path:

1. Privacy Officer confirms PHI scrubber posture for the affected
   tenant (the existing scribe parity contract enforces it; verify
   nothing in this incident bypasses the scrubber).
2. Clinical Director acknowledges the per-completion clinician
   disclosure will change from "sovereign GPU" to "Azure OpenAI".
3. Audit row with `reason: <sovereign_incident>` + signers recorded.
4. Same SQL UPDATE as above, flipping `llmBackend` to `azure_openai`.

## 4 — Visible clinician audit metadata

Every clinical-AI completion records the `RoutedModelExecution` on
`llm_interactions`. The clinician-visible disclosure surface (per the
existing scribe parity contract) displays:

```
AI generated by: <lane label>
Model: <deploymentRef>
Manifest: <modelVersion>
Generated at: <iso8601>
```

Failover is therefore clinically transparent — a clinician reading
yesterday's note sees yesterday's lane attribution and a clinician
signing today's draft sees today's lane attribution. No silent lane
swaps.

## 5 — Smoke verification after failover

Run the staging smoke against the affected clinic immediately after
failover:

```bash
./scripts/staging-smoke/ai-lane-smoke.sh "<clinic_id>" "<expected_lane>"
# - asserts /health returns 200
# - asserts /api/v1/ai/capabilities returns activeLane.lane == expected
# - asserts activeLane.health == "healthy"
# - asserts activeLane.deploymentRef matches the pinned reference
# - asserts promptPrefixHashSample is a 64-char sha256 hex
```

If any assertion fails, the failover is REVERTED via the same SQL
UPDATE (back to the prior backend) and the on-call escalates per
`docs/operations/incident-response.md`.

## 6 — Post-incident review

A failover event is a P2 incident regardless of customer impact.
Post-incident review per `docs/operations/incident-response.md` §6
includes:

- Lane that went unhealthy + root cause.
- Time from health signal to lane swap.
- Audit row referenced.
- Whether the smoke assertion passed on the failover destination.
- Any clinician-visible disclosure drift.
