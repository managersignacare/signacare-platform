# Repo Boundaries And Release Standard

This repo is the canonical Signacare product monorepo for the active web, API,
gateway, shared packages, and mobile clients. Split repos may exist for external
distribution or customer-specific packaging, but they are not the source of truth
for Signacare platform development.

## App And Package Boundaries

Runtime applications live under `apps/*`:

- `apps/api` — Express API, workers, migrations, and backend adapters.
- `apps/web` — clinician/admin web platform.
- `apps/emr-gateway` — integration gateway.
- `apps/mobile` — Sara clinician mobile client.
- `apps/patient-app` — Viva patient mobile client.

Shared packages live under `packages/*`:

- `packages/shared` — generated contracts, Zod schemas, DTO scaffolds, domain
  constants, and shared TypeScript utilities.
- `packages/ui-components` — shared UI components used by web surfaces.

No raw cross-app imports are allowed. An app must not import another app's
source through `../..`, `apps/...`, or other path tricks. Cross-app sharing must
flow through a package contract such as `@signacare/shared` or
`@signacare/ui-components`.

Packages must not import application source. If a package needs an app-owned
concept, move the contract into `packages/shared` first and then consume it from
the app.

## Generated Contracts

The backend API contract is generated from `apps/api/src/shared/swagger.ts` into:

- `packages/shared/src/generated/openapi.json`
- `packages/shared/src/generated/openapi.ts`

Regenerate with:

```bash
npm run contracts:generate
```

CI runs `npm run guard:api-client-contracts`, which regenerates the OpenAPI
artifact and fails if the committed contract is stale. Database row/DTO
contracts remain generated by:

```bash
npm run generate:types-from-migrations
npm run guard:generator-no-diff
```

Frontend and mobile route-call contracts are guarded by:

```bash
npm run guard:frontend-route-contract
npm run guard:dart-route-contract
```

## CI Standard

CI keeps safety broad and builds selective:

- Safety, security, migration, route, audit, and clinical guards still run
  broadly on every PR and protected-branch push.
- The production build job uses `scripts/ci/affected-workspaces.ts` to build
  only the app/package surfaces affected by the change.
- Boundary checks run in CI through `guard:cross-project-boundary` and
  `arch:depcruise`.
- The repository architecture contract itself is executable through
  `guard:repo-architecture-contract`.

The combined local architecture bundle is:

```bash
npm run guard:architecture-boundaries
```

## Scribe And Agentic Boundaries

The core medical scribe is the non-inferential clinical documentation pipeline.
It must not import optional agentic scribe modules or action-taking workflows.
Agentic scribe surfaces are optional, clinic-module-gated, fail-closed when the
module is not enabled, and must carry the canonical clinical AI disclaimer and
audit trail for clinician decisions.

Shared low-agency helpers, such as transcript-grounded action extraction, may
live in neutral shared modules. Agentic routes must not reach into `mcp/*`
scribe internals. This preserves the TGA/non-device posture of the core scribe
while allowing separately governed, human-in-loop agentic capabilities.

CI enforces this with:

```bash
npm run guard:scribe-agentic-isolation
```

## Deployment Standard

The active deployment path is Linux App Service plus managed Azure services:

- `deploy/azure/main.bicep`
- `deploy/azure/deploy.sh`
- `deploy/azure/preflight-linux.sh`
- `deploy/azure/post-deploy-smoke.sh`
- `.github/workflows/azure-deploy.yml`

The canonical application deployment pipeline is `.github/workflows/azure-deploy.yml`.
There must not be a second active deploy workflow for the same web/API estate.

The Azure deploy workflow authenticates with GitHub OIDC. Do not store or
reintroduce long-lived `AZURE_CREDENTIALS` service-principal JSON in GitHub
Actions. The Signacare tenant/subscription/OIDC client defaults are checked into
the workflow as non-secret identifiers and may be overridden by repo or
environment vars/secrets when a future tenant split requires it.

Development is local-first. Developers run and validate locally, commit to the
canonical repo, and push to GitHub. Azure deployment starts at staging; there is
no active Azure `dev` lane.

Deploy by digest. Staging builds create app images in CI, push them to ACR,
resolve each pushed image to `repo@sha256:...`, and configure App Service with
that immutable digest. Do not deploy mutable `latest` tags. Do not rebuild on
the target server. Do not manually patch staging or production images from a
laptop.

Staging is also a slot-based blue-green lane, not a direct live-site lane. The
workflow deploys staging images into the non-prod `next` slot, waits for API
`/ready=200`, proves `/version` on the slot against the new release manifest,
and only then swaps `next` into the live staging site. The release is not
considered live until post-swap smoke re-proves `/version` on the live staging
hostname.

Production is a promotion lane, not a rebuild lane. A prod dispatch must provide
the reviewed staging workflow run ID and staging image tag. The workflow
downloads that staging release manifest, imports the exact recorded image
digests into the production ACR with `az acr import`, writes a production
promotion manifest, and deploys the production App Service slot by the promoted
`prod-acr.azurecr.io/repository@sha256:...` references. The production manifest
records the source staging manifest checksum and source staging workflow run so
auditors can prove that production is running the artifact that passed staging.

Every Linux deployment must also produce an immutable release manifest:

- `artifacts/release/release-manifest.json`
- `artifacts/release/release-manifest.json.sha256`

The manifest records the commit SHA, workflow run, image digests, generated
OpenAPI hash, env/config contract hash, and migration head. Production promotion
manifests additionally record the staging source manifest checksum, staging ACR,
and staging workflow run. The active Azure workflow uploads the manifest as a
workflow artifact and stamps the same values into API App Service settings. The
API exposes those non-secret values at `/version`, and
`deploy/azure/post-deploy-smoke.sh` compares `/version` against the expected
manifest values before the deployment is considered healthy. Slot smoke proves
the release on the deployment slot before cutover, and post-swap smoke repeats
the same version proof against the live hostname after traffic is switched.

Release provenance must also be promotion-safe. Staging manifests created
outside GitHub Actions are break-glass only and must be marked
`manual_break_glass` plus `promotableToProd=false`. Production promotion is
allowed only from a reviewed staging manifest whose provenance is
`github_actions`, whose workflow is `Azure Deploy`, and whose manifest is
explicitly marked promotable to prod. This prevents a manual staging rescue
deploy from becoming an implicit production release source.

Production smoke is also observability-aware. The deploy workflow sets
`SMOKE_REQUIRE_OBSERVABILITY=true` for prod, and
`deploy/azure/post-deploy-smoke.sh` verifies that API App Service settings
contain Application Insights, OTEL, and both security/ops Slack webhook
settings. Staging can opt into the same fail-closed check, but staging remains
allowed to run unauthenticated smoke while smoke credentials are being prepared.

Drift detection is a separate, repo-owned control. The scheduled workflow
`.github/workflows/deployment-drift-audit.yml` logs into Azure with OIDC,
fetches the live API `/version` payload, compares that runtime proof with Azure
App Service `linuxFxVersion` digests and stamped `SIGNACARE_*` metadata, and
fails if the deployed environment has drifted away from its declared immutable
release. Staging is audited daily; prod remains manual until prod is activated.

Production database promotion is gated by release-control evidence. The deploy
workflow runs `deploy/azure/verify-database-release-controls.sh`, which requires
staging-clone migration proof, expand/contract proof, restore-drill proof, and
rollback-rehearsal or approved forward-fix-only proof before prod rollout.

This means a staging or production deployment must be able to answer:

- Which commit?
- Which image digest?
- Which migration version?
- Which config/env contract?
- Which workflow run?

The guardrail is `npm run guard:release-manifest-contract`. It prevents removal
of the manifest generator, `/version` runtime proof, Azure metadata stamping, or
post-deploy manifest comparison.

Long psychiatric interview scribe work belongs on the async AI queue, not the
synchronous `/api/v1/llm/clinical-ai` request path. The active architecture is
documented in `docs/architecture/async-ai-scribe-architecture.md`: submit with
`POST /api/v1/ai/jobs`, deliver progress over SSE (`ai-job-progress`,
`ai-job-complete`, `ai-job-failed`), and persist output before the UI reports
completion.

## Legacy Windows VM Lane

The Windows VM assets are retained for reference, forensic comparison, and
explicit customer-specific Windows-only decisions:

- `deploy/azure/main-windows.bicep`
- `deploy/azure/windows-vm/*`

They are not the active Signacare staging or production lane. Any future Windows
VM use requires an explicit decision record explaining why Linux App Service is
unsuitable and what additional hardening is required.
