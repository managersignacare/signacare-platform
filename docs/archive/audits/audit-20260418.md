# Audit 2026-04-18 — Schema drift inventory + Phase R R1 surfacing

This audit documents **all schema-drift findings** known as of 2026-04-18,
consolidated from:

1. Phase 0.7.5 c24 (Dec through Apr) — 62 confirmed runtime bugs (SD39–SD62)
   discovered while converting `.returning('*')` sites to explicit column lists.
2. Phase R R1 (2026-04-18) — 53 additional ghost-column writes surfaced when
   the new `check-code-writes-real-columns` CI guard ran for the first time
   against the current tree.

Every finding below is confirmed against the live Postgres via `psql \d`
or `information_schema` query, then cross-checked against the committed
`apps/api/src/db/schema-snapshot.json`.

Fixes land in one of three places:

- **Commits in-flight this session** (D1–D12a): each finding's fix is
  recorded in [../fixes/fixes-20260418.md](../fixes/fixes-20260418.md).
- **R1 guard exemptions** (53 ghost-column writes): marked inline with
  `// @code-columns-exempt: pre-R2 drift…` at each call site; listed in
  §3 below.
- **R2 consolidated baseline migration** (`20260701000000_baseline.ts`):
  every schema decision in the baseline (column names, NOT NULL, RLS,
  indexes) is the structural fix. See
  [../plans/master-plan-20260418.md](../plans/master-plan-20260418.md).

## §4. Structural root causes

Two structural causes produced all of the above:

### 4.1 No compile-time check of DB writes

Knex's `.insert<T>()` / `.update<T>()` types allow any `Partial<T>`. But
`T` in this codebase is almost always a hand-maintained interface or
`Record<string, unknown>`, neither of which reflects the DB schema. So
`.insert({ ghost_col: x })` compiles cleanly but is silently dropped at
runtime. R1's `check-code-writes-real-columns.ts` guard closes this
blind spot. Going forward, every `.insert/.update` is statically
cross-checked against the schema snapshot.

### 4.2 No enforcement of migration convention

The house style is "schema builder for simple DDL, `knex.raw` only for
operations the builder can't express" (RLS, triggers, functions,
partial/functional indexes, CHECK constraints, DML, partitioning,
extensions). There was no mechanical enforcement — only reviewer
discipline. This session's Phase 0.7.5 c24 migration-convention slip
(Claude writing `knex.raw(ALTER TABLE foo ADD COLUMN bar text)` in 4
migrations) demonstrated that discipline alone is not enough. R1's
`check-migration-convention.ts` guard closes this at CI time.

## §5. What R2 fixes structurally

The Phase R2 consolidated baseline migration
(`apps/api/migrations/20260701000000_baseline.ts`) is where every schema
decision above is made correctly from day one:

- Every column in §1 and §2 is either renamed in the baseline (code is
  right, previous DB was wrong) or added to the baseline (DB was right,
  previous code was writing a real field we just never added) or the
  code is corrected to stop writing a nonexistent column during R3's
  repository-translation-layer work.
- The 10 ghost tables (patient_tasks phantoms, care_plans family, etc.)
  are created with proper schema + RLS + indexes + triggers.
- Every `@migration-raw-exempt` annotation in the 6 legacy migrations
  ceases to matter because those files are archived (they ran before
  R2; they don't run again).

After R2 lands + the snapshot is regenerated:

- `guard:row-iface-drift` passes because every interface matches the
  canonical DB.
- `guard:code-columns` passes because every exempt-annotated site either
  had its code corrected (if the column was a typo) or the schema updated
  (if the column was intended).
- `guard:snapshot-freshness` passes because the baseline migration
  commits together with the regenerated snapshot.
